# Feature Specification: 団体メンバー・イベント管理ツール

**Feature Branch**: `001-member-event-mgmt`

**Created**: 2026-05-27

**Status**: Draft

**Input**: User description: "団体のメンバーやイベント管理をする簡易的なツール"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - メンバー登録・管理 (Priority: P1)

管理者がメンバーを登録・編集・削除し、メンバー一覧を確認できる。

**Why this priority**: メンバー情報はイベント管理の基盤となる最も重要なデータであり、これなくして他の機能は成立しない。

**Independent Test**: 管理者がメンバーを追加・編集・削除し、一覧に反映されることを確認することで独立してテスト可能。

**Acceptance Scenarios**:

1. **Given** 管理者がツールを開いた状態で、**When** メンバー追加フォームに氏名・連絡先・役割を入力して登録する、**Then** メンバーが一覧に表示される
2. **Given** メンバーが一覧に表示されている状態で、**When** 管理者がメンバーの情報を編集して保存する、**Then** 更新された情報が反映される
3. **Given** メンバーが一覧に表示されている状態で、**When** 管理者がメンバーを削除する、**Then** 一覧から該当メンバーが削除される
4. **Given** メンバーが登録されている状態で、**When** 管理者が一覧を表示する、**Then** 全メンバーの氏名・連絡先・役割・在籍状態が確認できる

---

### User Story 2 - イベント登録・管理 (Priority: P2)

管理者がイベントを作成・編集・削除し、開催予定および過去のイベント一覧を確認できる。

**Why this priority**: イベント管理は本ツールの主要機能の一つであり、メンバー管理の次に重要。

**Independent Test**: 管理者がイベントを作成・編集・削除し、一覧に反映されることを確認することで独立してテスト可能。

**Acceptance Scenarios**:

1. **Given** 管理者がツールを開いた状態で、**When** イベント作成フォームにイベント名・日時・場所・説明を入力して登録する、**Then** イベントが一覧に表示される
2. **Given** イベントが一覧に表示されている状態で、**When** 管理者がイベント情報を編集して保存する、**Then** 更新された内容が反映される
3. **Given** イベントが一覧に表示されている状態で、**When** 管理者がイベントを削除する、**Then** 一覧から該当イベントが削除される
4. **Given** 複数のイベントが登録されている状態で、**When** 管理者がイベント一覧を表示する、**Then** 開催日時順でイベントが表示される

---

### User Story 3 - イベント出欠管理 (Priority: P3)

管理者がイベントごとに参加メンバーを記録・確認できる（出欠の記録・閲覧機能を含む）。

**Why this priority**: 出欠管理はあると便利だが、「簡易的なツール」というコンセプトから、基本機能の後に検討すべき機能。

**Independent Test**: 特定のイベントにメンバーを参加登録し、参加者一覧に表示されることを確認することで独立してテスト可能。

**Acceptance Scenarios**:

1. **Given** イベントとメンバーが登録されている状態で、**When** 管理者がイベントに参加メンバーを登録する、**Then** イベントの参加者一覧にそのメンバーが表示される
2. **Given** 参加登録されたイベントがある状態で、**When** 管理者がそのイベントを確認する、**Then** 参加予定メンバーの一覧が表示される

---

### Edge Cases

- メンバーが0人の状態でイベントへの参加登録を行おうとした場合の動作
- 同じ氏名・連絡先のメンバーを重複登録しようとした場合の扱い
- 過去日付のイベントを新規作成しようとした場合の扱い
- 参加記録が存在するメンバーを削除しようとした場合の扱い
- 全データが空の初期状態での表示

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは管理者がメンバーを新規登録できなければならない（氏名、連絡先、役割、在籍状態を含む）
- **FR-002**: システムは既存メンバーの情報を編集・更新できなければならない
- **FR-003**: システムは管理者がメンバーを削除できなければならない
- **FR-004**: システムは全メンバーの一覧を表示できなければならない
- **FR-005**: システムは管理者がイベントを新規作成できなければならない（イベント名、日時、場所、説明を含む）
- **FR-006**: システムは既存イベントの情報を編集・更新できなければならない
- **FR-007**: システムは管理者がイベントを削除できなければならない
- **FR-008**: システムは全イベントの一覧を日時順で表示できなければならない
- **FR-009**: システムはメンバーとイベントのデータを永続的に保存しなければならない
- **FR-010**: システムはログイン認証なしで利用できるシングルユーザーツールとして動作しなければならない

### Key Entities

- **メンバー (Member)**: 団体に所属する個人。氏名、連絡先（メールまたは電話）、役割（例：役員・一般会員）、在籍状態（在籍中・退会済み）を持つ
- **イベント (Event)**: 団体が開催する集まりや行事。イベント名、開催日時、場所、説明、ステータス（予定・終了・中止）を持つ
- **参加記録 (Attendance)**: 特定のイベントに対する特定メンバーの参加状況（参加予定・欠席・未回答）を示す。メンバーとイベントを紐付ける

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理者がメンバーの登録から一覧確認まで3分以内に完了できる
- **SC-002**: 管理者がイベントの作成から一覧確認まで3分以内に完了できる
- **SC-003**: メンバー50名・イベント100件のデータを扱っても、一覧表示が3秒以内に完了する
- **SC-004**: 初めてツールを使う管理者が、マニュアルなしでメンバー登録・イベント作成の基本操作を完了できる
- **SC-005**: データの登録・編集・削除操作後、変更が即座に一覧に反映される

## Assumptions

- 対象ユーザーは小〜中規模の任意団体・サークル・コミュニティグループの管理者を想定する
- 初期リリース（v1）ではスマートフォン対応は対象外とし、デスクトップブラウザでの利用を前提とする
- 同時利用ユーザー数は少数（1〜数名）を想定する
- メンバー数の上限は数百名程度（大規模組織向けではない）
- 外部サービスや既存システムとの連携はv1の対象外とする
- データのインポート・エクスポート（CSV等）はv1の対象外とする
